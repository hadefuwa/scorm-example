How It Works

The workflow is staged on purpose so the source is inspected before anything is published.

The process now has initial evidence behind it: analysis, extraction, build, validation, and packaging reports are populated, and the repo contains real extracted and built files from the prototype run.

Word source Analyse Extract Build Validate Package SCORM

Block To Output

One tagged block becomes one controlled output.

The repo rules make the flow explicit: tag type sets the output type, nearby filename sets the target name, and instruction or teacher content is held back instead of being published into learner folders.

Real example

Block 17
Filename: CP4807-H1.htm
Classification: Homework
Intended output: HTML

Analysis identifies Homework 1 as a valid HTML block and resolves its filename directly from the source tags.

Resulting path

Extracted text: outputs/extracted/CP4807-H1.txt
Built output: outputs/html/CP4807-H1.htm
Validation result: no broken internal references detected in the built HTML file

Why The Stages Matter

The process protects learner-facing outputs from hidden ambiguity upstream.

The workflow now shows both control and evidence: warnings stay visible, teacher content stays separate, and SCORM packaging stops cleanly instead of being guessed from partial information.
