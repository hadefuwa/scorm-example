Source Tagging Rules

Every block must follow this exact pattern for the pipeline to process it correctly.

The analysis agent scans the source document for blocks that match a known tag pattern. A block that is malformed, missing a filename, or uses an unrecognised tag will be flagged in the analysis report and skipped during extraction.

Anatomy of a valid tagged block

<HTML>
Homework 1
<filename>"CP4807-H1.html"</filename>

Design a program that controls
two sets of traffic lights
at a junction.

</HTML>
Opening tag

Sets the output type for this block. Must be one of the recognised tags listed below. The AI uses this to decide what to build.

Display title

Optional. Appears as the heading in the built output. If omitted, the filename stem is used instead.

Filename tag

Required. Must be unique across the entire source document and must include a valid file extension.

Block content

The learner-facing material. Teacher notes and wrapper tags are stripped during the extraction stage.

Closing tag

Must exactly match the opening tag including case. <HTML> must close with </HTML> — not </html>.

Recognised tag types

The opening tag determines what the AI builds. Using an unrecognised tag causes the block to be skipped with a warning.

<HTML>

Learner HTML page

Builds a learner-facing web page. Used for homework, assignments, assessments, and any block that needs a browser-rendered output.

e.g. CP4807-H1.html
Assessment subtype becomes a SCO item in the SCORM manifest.
<worksheet>

Worksheet output

Builds a structured worksheet. Used for in-class activities or any block that needs a printable, downloadable format.

e.g. CP4807-1.docx
Not a SCO — does not become a trackable SCORM item.
<document>

Document output

Builds a reference or supporting document. Used for reading resources, guidance material, and supplementary content.

e.g. CP4807-CPDcert.pdf
Supplementary resource only — not tracked by the LMS.

Filename Rules

Filenames must be unique, correctly typed, and include the right extension.

The filename tag is how the pipeline knows where to write each output. An incorrect or duplicated filename breaks the chain for that block only — the rest of the source continues to process.

Must be unique across the whole source document

No two blocks can share a filename. Duplicates are flagged in the analysis report and the second block is skipped.

Must include a file extension

The extension tells the pipeline which folder to route the output to. Without one the file cannot be placed or built.

Must be wrapped in double quotes inside the tag

The value must be quoted: <filename>"CP4807-H1.html"</filename>. Unquoted filenames may not be parsed correctly.

Extension should match the tag type

An <HTML> block should use .html or .htm. A mismatch creates a routing warning in the analysis report.

SCORM Source Requirements

Five decisions must be defined before SCORM packaging can run.

The pipeline reads these from the source document or a companion definitions file. If any are absent, packaging stops safely and the missing items are listed in the packaging report. None of these can be guessed or defaulted.

1

SCORM version

Specify 1.2 or 2004. The manifest schema, API calls, and data model differ between versions. This cannot be derived from the content.

e.g.scorm-version: "1.2"
2

Launch file

The filename of the first page an LMS should open. Must be one of the built HTML outputs and must already exist before packaging runs.

e.g.launch: "CP4807-H1.html"
3

Tracking model

Whether the LMS tracks completion, score, or both. Determines which SCORM API calls are written into the SCO and how the manifest declares them.

e.g.tracking: "completion-only"
4

Completion model

How the LMS decides a learner has finished. Common options are page-view, quiz-pass, time-based, or manual. Must be explicitly chosen by the author.

e.g.completion: "page-view"
5

Navigation rules

Whether learners can move freely between items or must follow a linear sequence. Affects the manifest organisation and item ordering in the LMS.

e.g.navigation: "free"

All five must be defined. The packaging prompt explicitly checks for each one and stops with a clear error listing what is missing. Partial definitions are not used — the whole packaging stage is deferred until all five are present.

Prompts And Rules

The prompts line up directly with the stage model.

Prompt files

  • `prompt-01-analyse.md` finds blocks, filenames, warnings, and SCORM gaps.
  • `prompt-02-extract.md` creates the staged block files.
  • `prompt-03-build.md` creates learner-facing outputs.
  • `prompt-04-package-scorm.md` stops packaging if the required SCORM decisions are missing.

Rules files

  • `tag-rules.md` defines block recognition and filename binding.
  • `output-rules.md` defines output behavior and naming.
  • `scorm-rules.md` defines minimum requirements and stop conditions.
